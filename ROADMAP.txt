# SpectralRenderer - Roadmap

Long-form planning doc. Companion to `CLAUDE.md` (which covers *how* to work
in the codebase). This doc covers *what* to build next and why.

Update this as phases close or priorities shift. When in doubt, add a note
rather than rewrite -- context rot beats missing context.

---

## Current status (April 2026)

Phase 17 + 30 + infrastructure complete. The renderer has:
- Spectral path tracing (hero wavelength, 380-780nm, CIE 1931 2deg observer)
- CPU path via Embree 4, GPU path via OptiX 9 / CUDA 12.6
- UsdPreviewSurface materials with spectral extensions (dispersion, thin-film,
  metal BRDFs, SSS, diffraction gratings, fluorescence, Beer-Lambert)
- Volumes via OpenVDB / NanoVDB, multi-volume support, fire & explosions
- HDRI / studio / sun+sky lighting
- Shadow catcher material (AOV-style compositing surface)
- Artistic drafting overlay (6 styles incl. pencil sketch & topographic)
- SpectralRenderIop frontend for Nuke 17 node graph
- Hydra render delegate stub for Hydra viewer path (Phase 1 only)
- OpenSubdiv Catmull-Clark for displacement and smooth surfaces
- SpectralMeshProperties per-mesh override node
- GPU object motion blur (Phase 30) via OptiX 2-keyframe motion GAS
- ScanlineRender compatibility mode with correct env lighting, double-sided
  shading, hard + soft shadows
- Per-mesh overrides via prim-path registration: visible, castsShadows
  (2026-04-19)

Stack: Windows 11 + VS2022, Nuke 17, CUDA 12.6, OptiX 9.0, RTX 5060 Ti.

## Near-term backlog (next 1-3 sessions)

Ordered roughly by value. Nothing here is committed; reorder as needed.

### DAG-walk registry cleanup

Destructors on SpectralSurfaceOp / SpectralDraftingOp / SpectralShadowCatcherOp
/ SpectralMeshPropertiesOp DON'T fire when the user deletes the node in Nuke
-- Nuke keeps the Op alive on the undo stack. Registry entries persist.

Fix: in SpectralRenderIop::_validate, walk the live Op tree, collect all
names, erase registry entries whose key is not in that set. O(N*M) but N
and M are small. Belt-and-braces: the existing destructors are kept for
session-exit and rare reload paths.

### SpectralVelocity node (Phase 33) [REPLACES CHAIN-WALK OBJECT MBLUR]

Houdini-style per-vertex velocity attribute (`v` or `primvars:velocities`)
as the signal for object motion blur, instead of the chain-walk transform
inference that we shelved in session 2026-04-19. Benefits:

- Explicit input, no ambiguous transform inference
- Composes naturally with deformation (cloth sim, morphs, rigs)
- Simpler to reason about and test
- Survives more complex scene topologies

New Op: `SpectralVelocity` -- takes geometry input, generates per-vertex
velocity vectors from a connected image (2D velocity channel) or from
a knob-driven analytic function. Writes `primvars:velocities` onto the
prim. SpectralRenderIop consumes that primvar when building the close-
time mesh, producing deformation blur on top of any existing transform
blur.

### Shadow Catcher v2 (Phase 24.1)

Current `SpectralShadowCatcherOp` exposes three knobs: `shadow_intensity`,
`shadow_color`, `self_shadow`. That's enough for basic compositing but is
behind Arnold/Redshift/Cycles feature-for-feature, and several common
user requests go unserved.

Research (2026-04-19 session) collected feature inventory across Arnold
shadow_matte, Redshift Matte object-tag, Cycles 3.0+ object flag, and
ProRender. Synthesis of user requests on Blender devtalk, Octane forums,
SideFX forums, and VFX workflow articles. Priority-ordered plan:

**Phase A - AOV plumbing (foundational).** Add a dedicated shadow-alpha
AOV output channel filled at shadow-catcher hits. At each hit, accumulate
`1 - visibility` into a separate buffer exposed as an extra channel in
the output. Also add a reflection-catch AOV at same time: same hit point
but accumulating the specular contribution that would have lit the
catcher. Single most valuable comp-side addition. Mostly plumbing --
AOV buffer infrastructure already exists for other channels.

**Phase B - Per-knob granularity on the node.** Split the existing knob
set to match Arnold/Redshift patterns:
- existing: `shadow_intensity`, `shadow_color`, `self_shadow`
- new: `catch_reflections` (bool), `reflection_intensity` (float),
  `catch_indirect` (bool), `indirect_intensity` (float),
  `background_mode` (enum: transparent / scene_env / explicit)

Half-day UI plumbing once Phase A lands.

**Phase C - Indirect participation (the Cycles-3.0 upgrade).** Let the
catcher's real material participate in secondary rays: a chrome car
above a green shadow-catcher ground should reflect green in its
underside. Changes what "hitting a shadow catcher" means at the
integrator level -- surface behaves like a normal surface except for
primary-camera-ray visibility. Biggest change of the three, but the
feature that separates "works OK for obvious shots" from "holds up in
a 4K A/B against a plate." Both CPU and GPU.

**Phase D - Per-light shadow AOVs.** One float channel per light for
shadow density, enabled via knob (disabled by default -- adds buffer
memory cost). Most-requested feature from VFX compositors: lets them
rebalance key-to-fill ratio or push a rim light in comp without a
re-render. Requires Phase A infrastructure.

**Phase E - Reflection catcher as separate flag.** Following ProRender,
allow the node to toggle between catching shadows, reflections, or
both. Requires Phases A + C.

### Finish SpectralMeshProperties wiring

Prim-path registration landed (session 2026-04-19). Now the following
registered-but-unused knobs can be wired safely without cross-mesh
leakage:

- `doubleSided` -- backface culling flag per mesh. CPU integrator and GPU
  kernel both need to check this before rejecting a back-facing hit.
- `receivesShadows` -- per-mesh flag for skipping shadow tests when
  shading this surface (mirrors castsShadows but on the receiver side).
  Useful for CG reflections of set pieces that shouldn't be shadowed.
- `orientation` -- winding-order flip. Already implicitly supported via
  `flipNormals`; merge the concepts or make `orientation` override.
- `purpose` -- USD render/proxy/guide. Affects inclusion in render vs
  viewport vs bbox.

### Volumetric refraction

Glass containing smoke / fire / fog. Currently refraction rays see the
volume as empty space. Needs the refraction path to call the volume
integrator between hits, not just surface-to-surface.

### Emission from volumes lighting geometry

Currently volume emission (fire, explosions) is visible but does not
contribute to geometry lighting. Needs virtual-light extraction from
emissive voxels, similar to how HDRI virtual lights already work.

### Full path tracing

Currently a hybrid direct-lighting + limited-bounce indirect. Unified
path tracing with MIS would simplify the integrator, improve convergence
for complex lighting, and match industry expectations.

### SpectralSurface PBR expansion (Phase 21)

- Anisotropic roughness (brushed metal, hair, fabric)
- Coat normal maps (independent of base normal)
- Energy compensation for rough conductors
- Better Fresnel for non-metals (Schlick -> exact Fresnel, optional)

### Prebuilt material library (Phase 18)

Curated spectral material presets as a picker menu:
- Common metals (gold, copper, silver, aluminium, chrome) -- already in code
  as `metalType` enum, expose as a proper material browser
- Glass variants (crown, flint, sapphire, diamond, quartz) with correct
  Abbe numbers
- Skin (with matching SSS profile)
- Common plastics, ceramics, woods

## Technical debt / known issues

Tracked here so we don't forget. Each line is roughly the shape of an issue.

- **SpectralMeshProperties path matching**: `size() == 1` fallback is brittle;
  breaks with >1 modifier node. See "Prim-path registration" above -- this is
  the blocker for several other features.
- **SpectralDrafting motion blur**: drafting overlay is a post-process pass,
  not integrated into the motion-blur time sampling. Wireframes on moving
  objects won't smear with the blur. Acceptable for current scope.
- **PointInstancer + MeshProperties**: PointInstancer path doesn't consult
  the MeshProperties registry at all. Separate fix once prim-path reg lands.
- **Volume compositing on CPU**: volumes currently render behind geometry
  on the CPU path. GPU is correct. Fix the CPU path to match.
- **Neutral white balance**: only applied in non-scanlineCompat spectral mode.
  Tooltip says so but this is a quirk, not a feature. Consider a dedicated
  white-balance handling that works in all modes.
- **Compat shadow noise**: soft shadows use N=4 samples per pixel per light.
  Sufficient for still frames but noisy in sequences with high-softness
  settings. Promote SOFT_SHADOW_SAMPLES_* to a knob when someone complains.
- **32-material cap on noShadowCastMask / shadowCatcherMask**: bitmask
  limits us to 32 unique materials for each flag. Scenes with hero characters
  + dozens of props could brush this limit. Log warning exists; upgrade to
  wider type (uint64_t or a proper per-material flag) when needed.
- **SpectralMeshProperties purpose menu needs validation** (new, 2026-04-20).
  apply_meshprops_fields wired up doubleSided / orientation / purpose, but
  purpose (default / render / proxy / guide) hasn't been tested end-to-end.
  The intended behaviour: when purpose=proxy or purpose=guide, the mesh is
  skipped from the ray-traced render (still shows in the 3D viewer). The
  mesh-skip `continue` is in SpectralRenderIop.cpp around line 3286 guarded
  by `meshPropsHasEntry && (meshPropsPurpose == 2 || meshPropsPurpose == 3)`.
  To validate: (1) confirm the 'mesh props for ...' log shows the purpose
  value flipping when you toggle the enum; (2) confirm 'SpectralRender:
  skipping mesh X (purpose=proxy)' logs when set to proxy; (3) confirm the
  mesh visually disappears from the rendered frame but remains in the
  viewer. Also: decide whether purpose=1 (render) should imply anything
  different from purpose=0 (default). USD treats 'render' as 'always
  include', 'default' as 'parent decides', so currently both render.
- **[RESOLVED 2026-04-21] SpectralMeshProperties doubleSided on CPU**
  (was tracked as "doubleSided intermittent"). Original triage blamed
  GPU stamping races and kernel shade-path gaps. Actual cause: GPU was
  fine. CPU had two unrelated bugs.
  Truth table from inside-sphere + card-backside tests:
    inside sphere, doubleSided=true : GPU lit, CPU BLACK (bug)
    inside sphere, doubleSided=false: GPU black alpha=0, CPU black alpha=0 (ok)
    card backside, doubleSided=true : GPU lit, CPU lit (ok)
    card backside, doubleSided=false: GPU black alpha=0, CPU black ALPHA=SOLID (bug)
  Bug 1 (inside-sphere self-shadow): shadow ray from backface-flipped N
    offset = hitPos + N*0.01 pushed shadow origin INTO the sphere, then
    the shadow ray exited through the far shell wall and registered that
    exit as occlusion. Two-part fix:
      - Sign-flip origin offset: `hitPos + N * (NdotL >= 0 ? 0.01 : -0.01)`
        at lines 1830 (direct shadow) and 1873 (volume shadow). Matches the
        pre-existing idiom on line 508 of the compat shadow path.
      - Shell-passthrough in the shadow loop: if the blocker is
        doubleSided AND `L . faceNormal > 0` (shadow ray exiting the
        blocker's back), step past and continue. Matches GPU behavior.
        Trade-off: internal self-shadowing of concave doubleSided geometry
        is suppressed (a cave interior lit by an internal light won't
        shadow itself). Accepted.
  Bug 2 (CPU alpha on single-sided backface): _ShadeSpectral returned 0
    radiance via `if (!mat.doubleSided) return 0.f` but the outer RenderFrame
    loop still wrote alpha from material.opacity. Fix: at the top of the
    primary-hit-processing block, if `rayDir . faceNormal > 0` AND
    material is single-sided, set `hit.tri = nullptr`. The downstream
    hit.valid() guards then cascade: no radiance, no alpha, no depth, no
    objId write. Matches GPU exactly (GPU culls at intersection stage).
  Patches: apply_cpu_doubleside_diag (tracers), apply_cpu_doubleside_fix
    (origin offset), apply_cpu_doubleside_shell_fix (shell passthrough),
    apply_cpu_backface_cull_fix (primary-hit invalidation).
  Leftover possible-latent: 4 other instances of the `hitPos + N * 0.01f`
    pattern remain unmodified (line 1356 AO, 2442 indirect bounce, 2748
    transmission). They may self-occlude similarly when rays originate
    from backface-flipped surfaces. Parked pending user-visible symptoms.
- **Surface-shader CPU/GPU parity audit gaps** (new, 2026-04-20).
  Audit completed on 2026-04-20 found that GPU shade path matches CPU
  for base colour, metallic, roughness, IOR, opacity, Abbe dispersion,
  thin-film, measured metals, clearcoat, Beer-Lambert absorption,
  grating diffraction, fluorescence, subsurface scattering (random walk),
  texture blend, shadow catcher, and dielectric transmission. Four gaps
  remain. Fix each as a standalone follow-up: (a) roughnessTexId -- CPU
  samples texture green channel to override roughness per-pixel; GPU
  ignores the field entirely. Needs GPUMaterial addition, host upload
  + matCheck coverage, kernel sample and override before BSDF eval.
  (b) metallicTexId -- CPU samples blue channel; GPU ignores. Same
  shape as (a). (c) Spectral emissive -- CPU adds mat.SpectralEmission
  (lambda) to radiance at every shade; GPU only handles emissive in
  scanlineCompat RGB path, spectral shade block has no emissive
  contribution. (d) Bump-map tangent frame -- CPU builds UV-gradient
  derived T/B from triangle vertex data; GPU uses an arbitrary
  "up x N" fallback. Bump pattern rotates differently per face. Fix
  needs triangle vertex positions plumbed through OptiX SBT
  (HitGroupData or per-triangle ptr in params).
- **Volume shader CPU/GPU parity audit needed** (new, 2026-04-20).
  Counterpart to the surface-shader audit. Scope: compare every field
  of SpectralVolumeMaterial / SpectralVolume against GPUVolume in
  SpectralGPUParams.h, and compare CPU shade path in SpectralIntegrator
  (volume section) against the kernel's volume shading. Known GPU
  fields that exist on GPUVolume: density/temp/flame tex, bbox,
  extinction/scattering/densityMult, phase (HG dual-lobe + Cornette-
  Shanks Mie), emission, powder, scatter colour, step size, jitter,
  shadow steps, shadow density, quality, adaptive step, render mode
  (Lit/Grey/Heat/Cool/BB/Expl), flame temperature range, core glow,
  Cherenkov, chromatic extinction, multiple-scattering approximation,
  grid noise, transform, grid mixer. Likely suspects before starting:
  the CPU path might have spectral-emission temperature blackbody
  handling that GPU lacks (or vice versa); volume-colour absorption
  across wavelengths may diverge; shadow-density sampling step counts
  may differ. File findings as a separate known-issues entry once the
  audit lists discrepancies.
- **HDRI "visible in background" checkbox does not render the BG** (new,
  2026-04-20). apply_hdri_visible patch added the knob on SpectralEnvLight
  plus the full plumbing: host member, SpectralLight.visibleInPrimary field,
  stamp at both hdriDome sites, CPU primary-miss gate, GPU LaunchParams
  fields, GPU miss-shader HDRI sampling. Compiles and runs, but toggling
  the checkbox does not change what the user sees. Suspects in priority
  order: (1) checksum miss -- _envVisibleBg change may not invalidate
  the GPU params upload (verify with a log line in the lightCheck block);
  (2) the extract loop populates from the first Dome that HAS visibleInPrimary
  set, so if the user's scene has multiple dome lights and the flag is on
  the wrong one, code takes the first hit; (3) CPU path: verify the flag
  is making it through SpectralLight copy construction (the light is
  copied into `scene._lights` by value); (4) the envTexId the kernel
  samples may be pointing at the wrong texture (HDRI tex upload order
  could have shifted since the field was added). Triage plan: add an
  SLOG at each stamp site (Iop, host extract, LaunchParams fill) printing
  envVisibleBg + envTexId, then toggle the knob and watch the log
  transitions.
- **Nuke crash when clicking the HDRI file-path knob on SpectralEnvLight**
  (new, 2026-04-20). User reported a crash after interacting with the
  file browser on SpectralEnvLight -> HDRI -> file. No trace captured yet.
  First suspects: the File_knob browse callback pathing (unicode? network
  share?), the existing cold-load-to-texId path (SpectralRenderIop.cpp
  ~line 8452 `el->hdriFile && strlen(el->hdriFile) > 0`), or a LoadTexture
  reentrancy during active render. Next time it repros: run Nuke from a
  cmd prompt and look for stderr, or attach Visual Studio debugger. A
  defensive try/catch around the file-path reads + LoadTexture call would
  reduce blast radius if the exception path is the cause.

## Open design questions

These need more thought before turning into tasks.

### Hydra delegate scope

How far do we take `HdSpectralRenderDelegate`? Full parity with the Nuke
Iop path means duplicating scene conversion, material binding, light
enumeration, AOV buffer management. Currently it's a Phase-1 stub doing
normal-shaded geometry.

Options:
1. Keep as a stub, mark as "out of scope", don't maintain
2. Bring to Phase-4 feature parity (materials, lights, volumes) and use it
   as the viewport path for Nuke 17's Hydra-based renderer
3. Full parity including GPU path -- effectively two frontends for the
   same core renderer

Gut feeling: (2) is worth the work because Nuke is moving to Hydra for
viewport rendering and being absent from that is bad positioning. But it's
a quarter of engineering at minimum.

### Out-of-core rendering (Phase 19)

Scenes larger than VRAM. Two approaches:
1. **Tile-based scene streaming** -- bring geometry in/out as GAS tiles,
   retrace pixels that miss. Needs careful BVH design and significant OptiX
   restructuring.
2. **Managed memory / UVA** -- let CUDA handle paging. Simpler but much
   slower under realistic conditions.

VFX production workflow normally pre-loads everything, so this mostly
matters for very large single assets (multi-million-poly environments,
high-res VDB caches). Low priority unless a specific user needs it.

### Linux build (Phase 31)

GCC/Clang on Ubuntu 24. Needed for farm rendering in production pipelines.
Gotchas already surfaced in earlier work:
- `_USE_MATH_DEFINES` is MSVC-specific; use literal `3.14159265f`
- OpenVDB linking is fiddlier (vcpkg vs system packages)
- GLEW linking fallback chain already in CMakeLists
- CUDA/OptiX paths differ

Not hard, just tedious. Worth doing before public release.

### Multi-GPU (Phase 23)

Two GPUs in one box is increasingly common. Simplest model: tile the image
across GPUs, composite on download. More complex: share GAS and texture
memory across GPUs via NVLink / P2P.

Not urgent -- single-GPU on a 5060 Ti or better hits interactive speeds
on most scenes -- but worth designing for before API choices lock it out.

## Things we will not build

Keeping these explicit prevents scope creep conversations.

- **A node editor / material graph UI**: Nuke's node graph is the UI.
- **GPU compiler replacement**: OptiX/CUDA is fine.
- **Full BSSRDF / volumetric SSS**: the current MPFR-less Christensen-Burley
  SSS is good enough for VFX work. Exact BSSRDF is research territory.
- **A denoiser of our own**: OIDN and OptiX denoiser cover the need.
- **Chain-walk object motion blur**: shelved this session in favour of
  SpectralVelocity (Phase 33). The chain-walk approach requires inferring
  transforms from the Op tree, which doesn't compose with deformation and
  had too many edge cases.

## Known issues / parked work

Bugs and rough edges that are real but not blocking. Documented here so
we don't re-diagnose them next session.

### Viewport tumble is slow on dense geometry

Scenes with >~50k triangles cause the 3D viewport to stutter during
tumble. Native Nuke viewport handles the same scenes smoothly, so this
is our code, not a driver / hardware limit.

Cause confirmed via tracers: `draw_handle` iterates all triangles in
four places per frame using OpenGL immediate mode (`glBegin(GL_TRIANGLES)`
+ `glVertex3f` per vertex). At 120k tris and 60fps that is ~21M
`glVertex3f` calls per second, which the driver cannot keep up with.
Sites:
  - Scene bounds for shadow-map lightVP (full iteration to compute AABB)
  - Shadow-map depth pass (the dominant single cost)
  - Reflection pre-pass (when `vp_geo_reflections` is on)
  - Main geometry draw

Things tried that did not fix it:
  - `vp_enable_shadows` knob added this session; default off. Gates only
    the shadow-map pass. Shrinks the problem but does not solve it --
    the other three sites still iterate 120k tris immediate-mode.
  - Part B of a VBO migration (`.bak_tumblefixB` backups -- now reverted
    in place). Migrated all four sites at once to `glDrawArrays`. Broke
    shading (geometry went near-black) without fixing perf. Too ambitious
    for one patch.

Infrastructure that stays in place for a future proper fix:
  - `_EnsureGeoVBO()` helper (see SpectralRenderIop.cpp)
  - `_glGeoVBO`, `_glGeoMeshRanges`, `_glSceneMin/Max`, `_glGeoVBODirty`
    members (see SpectralRenderIop.h)
  - `TumbleDiag:` tracers in `append`, `_validate`, `draw_handle`

Next attempt should migrate ONE site at a time (test, confirm, continue),
starting with the shadow-depth pass since it has no material uniforms,
normals, or UVs -- just positions. If that works, move to reflection,
then main draw. Watch out for GL client-array state bleed between the
immediate-mode light gizmos/handles and the VBO sites -- a reasonable
guess is that Part B's failure was client-state contamination.

Alternative: write a custom viewport path (VBO-only from the start,
own shader) instead of extending the existing ad-hoc draw_handle mess.
Bigger scope but cleaner result.

### Nuke crashes on exit

As of ~2026-04-19. Has not been triaged -- needs own session.

Likely culprits (in rough probability order):
  - Static `std::unordered_map` registries in the shader/geom ops
    (`SpectralSurfaceOp`, `SpectralWireframeOp`, `SpectralShadowCatcherOp`,
    `SpectralMeshPropertiesOp`) accessed during Nuke's Op teardown in
    an order-of-destruction-dependent way. Static-destruction order
    bugs are classic crash-on-exit offenders.
  - OptiX resources not released on plugin unload (pipeline, SBT,
    module, context, disk cache lock).
  - CUDA context outliving something it points to.

To triage:
  - Reproduce: does it crash on exit from a fresh empty .nk? From a .nk
    with SpectralRender nodes but never rendered? After one render? After
    many renders?
  - Compare: rename SpectralRender.dll to disable the plugin; does Nuke
    exit cleanly?
  - Stack: attach debugger, grab stack at crash, check whether it's our
    code or Nuke's (or a driver DLL).
  - Valgrind / ASAN if feasible.

### GPU ambient occlusion + dedicated 'ao' output layer -- RESOLVED 2026-04-21

GPU path for AO AOV: previously AO ran on CPU only (via
SpectralIntegrator::ComputeAO) regardless of the render device. Scenes
with GPU device + AO enabled paid the CPU-AO cost on top of GPU beauty.
Also AO output lived under 'other.ao', harder to find in the Viewer
channel dropdown than a top-level layer.

Shipped end-to-end:

  1. __raygen__ao kernel program (SpectralGPUKernel.cu): uses the
     existing makeRay() helper (shared with __raygen__spectral so
     primary rays match across perspective / spherical / DOF /
     motion-blur camera modes), traces primary ray via shared
     closesthit, reconstructs flipped-to-camera normal from payload,
     builds TBN, launches aoSamples cosine-hemisphere AO rays with
     CULL_BACK_FACING_TRIANGLES, writes 1/aoSamples per unoccluded
     ray into aoBuffer.
  2. Host pipeline (SpectralGPU.cpp): pipeline link extended from
     3 -> 4 program groups (AO raygen added). Separate _sbtAO with
     the AO raygen record; shares miss + hitgroup records with the
     main _sbt. Device buffer _d_aoBuffer sized W*H floats, reused
     across frames, resized on format change.
  3. Integrator wrapper (SpectralIntegrator::ComputeAOGPU): mirrors
     the RenderFrameGPU pattern. Gets GPU, BuildAccel, calls
     SpectralGPU::ComputeAO, falls back to CPU ComputeAO on any
     failure path (no GPU, accel build fail, launch fail).
  4. Iop dispatch (SpectralRenderIop.cpp): AO call site branches on
     _renderUseGPU.
  5. UI polish: new 'Ambient occlusion' checkbox gates samples +
     radius sliders. Defaults tuned to samples=16 and radius=2.0
     (earlier default radius=5 was too large for typical scene
     scales and gave mostly-unoccluded output). Backward-compat
     sync in knob_changed panel-open: scenes with existing
     _aoSamples > 0 auto-tick the checkbox.
  6. Output layer: AO now surfaces as 'ao.red'/'ao.green'/'ao.blue'
     (top-level layer, scalar replicated into R/G/B), replacing
     'other.ao'.

Critical bug + fix (SBT hitgroup):
  _sbtAO was populated in Initialize() with raygen + miss records,
  but hitgroupRecordBase/Stride/Count were NEVER set. Those fields
  get populated for _sbt inside BuildAccel() at line ~943-945; I
  mirrored the raygen+miss setup in Initialize for _sbtAO and missed
  BuildAccel's hitgroup step. On first AO ray hit, OptiX dereferenced
  a zero hitgroupRecordBase -> CUDA "illegal memory access". The
  error poisoned the CUDA context so EVERY subsequent optixLaunch

### SpectralMeshProperties doubleSided CPU fixes -- RESOLVED 2026-04-21

GPU render of doubleSided materials behaved correctly; CPU had two
distinct bugs + one asymmetry vs GPU. Original triage wrongly blamed
GPU. Real picture came out via DSDIAG tracers:

  Bug 1 -- inside-sphere + doubleSided=true rendered BLACK on CPU.
  Root cause: primary ray hits a backface, we flip N to face the
  camera (inward in this case), shadowOrigin = hitPos + N*0.01
  pushes INTO the sphere volume, shadow ray then exits through the
  far wall which IS geometry -> false self-occlusion -> zero direct
  lighting. Two-part fix:
    (a) Sign-flip offset: hitPos + N * (NdotL >= 0 ? 0.01 : -0.01)
        so the shadow origin rides on the correct side of the shell
        (apply_cpu_doubleside_fix.py).
    (b) Shell passthrough in the shadow loop: if the blocker surface
        is itself doubleSided and L dot faceNormal > 0 (shadow ray
        exiting a shell rather than entering geometry from outside),
        skip the occlusion contribution. Completes the fix for
        closed doubleSided shells (apply_cpu_doubleside_shell_fix.py).

  Bug 2 -- card-back + doubleSided=false rendered alpha=solid on
  CPU even though RGB=0. GPU correctly gave alpha=0 because its
  backface cull happens at the OptiX intersection level (ray filtered
  before any shade). CPU previously reached shade with hit.valid(),
  returned zero radiance, but alpha=1 from the presence of a hit.
  Fixed by culling at primary-hit entry in RenderFrame: if
  rayDir dot faceNormal > 0 AND !mat.doubleSided, set hit.tri =
  nullptr so downstream hit.valid() guards cascade as "no hit"
  (apply_cpu_backface_cull_fix.py). Matches GPU's at-intersection
  semantics exactly.

Testing matrix confirmed: card front/back x doubleSided on/off x
inside-sphere -> all 6 cases match GPU output. DSDIAG tracers
retired via apply_doubleside_cleanup.py.

Paranoia check on 3 other `hitPos + N * 0.01f` offset sites (AO
main, AO fallback, bounce): verified safe -- those use cosine-
hemisphere sampling around flipped N, so the offset direction
always agrees with the sample direction. No self-shadow geometry
like the shadow loop. AO radius naturally filters far-wall
self-hits. Photon site is dead code (caustics disabled).

### SpectralSurface preset UI refresh -- RESOLVED 2026-04-20

Was: picking a preset from the dropdown would update the rendered
material but sliders on the panel wouldn't redraw. Variations: presets
worked for the first pick or two then silently broke, especially jade
and spectral-category items. Symptom was NOT "bound member didn't
change" -- the render output showed the new preset values correctly --
but the panel widgets stuck on old values AND after a few picks the
preset knob stopped even firing knob_changed.

The earlier (wrong) diagnosis assumed Nuke's change detection was
skipping widget invalidation when members were pre-written. That was
BACKWARDS. The truth came out via diagnostic tracers: after the first
preset pick, subsequent picks fired knob_changed on a DIFFERENT Op
instance than the one displayed. The Op got rebuilt mid-callback.

Actual root cause: two things the rewrite had wrong vs SpectralVolume-
Material (which always worked reliably):

  1. Missing `return 1;` after preset application. Without it,
     knob_changed falls through to `ShaderOp::knob_changed(k)` which
     runs Nuke's default handling. For reasons specific to Nuke 17's
     ShaderOp machinery, that default handling rebuilds the Op when
     a ShaderOp's knob set changes, which invalidates `this` and
     routes future callbacks to a fresh instance.

  2. Writing members via `set_value` only, without preceding direct
     member assignment. Every `set_value` where the bound value
     differs fires a deferred `knob_changed` for that pushed knob.
     When we push ~20 knobs in a row this cascades, and Nuke's
     instance swap happens somewhere in the cascade.

Volume pattern (which works) inverts both:
  - member = local; (direct write first)
  - if (Knob* k = knob(...)) k->set_value(v);  (now a no-op, suppresses
    the callback storm because Nuke sees bound value already matches)
  - return 1;  (after preset applied, don't fall through)

Fix applied in `apply_surface_preset_fix.py` (2026-04-20) plus unified
master dropdown UI from `apply_surface_preset_unify.py`. Diagnostic
tracers and cleanup in `apply_surface_preset_diag*.py`.

Lesson for future Nuke plugin work:
  - ShaderOp::knob_changed path may rebuild the Op under default
    handling. Always `return 1` when you've fully handled a knob.
  - For panel-pushing code (presets, auto-populate, etc.), write the
    member directly FIRST then call set_value -- Volume's pattern.
    set_value as the ONLY write path triggers callback storms that
    can end up swapping the Op out from under you.
  - Diagnostic tracers with `this=%p` snapshot on knob_changed entry
    are invaluable -- if the pointer changes mid-session, you're
    chasing an instance-lifetime issue, not a value-handling one.

Associated backups (.bak_surfunify, .bak_surfdiag, .bak_surfdiag2,
.bak_surffix, .bak_surfclean) can all be discarded once confirmed
stable.
